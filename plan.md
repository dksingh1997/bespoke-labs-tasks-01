# Plan: New Code Formatter Tasks

## Overview

5 new Harbor tasks that challenge agents to implement deterministic code formatters from scratch in the same language as the target. Each task requires byte-exact matching against the language's reference formatter across ~200 test cases.

## Tasks

| Task | Target Language | Implementation Language | Reference Formatter | Cases |
|------|----------------|------------------------|--------------------|----|
| `code-formatter-go-go` | Go | Go | `gofmt` | 200 |
| `code-formatter-rust-rs` | Rust | Rust | `rustfmt` | 200 |
| `code-formatter-py-py` | Python | Python | `black` | 200 |
| `code-formatter-c-c` | C | C | `clang-format` (LLVM) | 200 |
| `code-formatter-java-java` | Java | Java | `google-java-format` | 200 |

## Task Structure (per task)

```
code-formatter-<lang>-<lang>/
├── instruction.md              # What the agent sees
├── task.toml                   # Harbor task config (difficulty, timeouts, resources)
├── README.md                   # Full documentation
├── generate_cases.py           # Script to generate test cases from reference formatter
├── environment/
│   ├── Dockerfile              # Language-specific environment
│   └── formatter/              # Skeleton codebase (copied to /app)
│       ├── main.<ext>          # Entry point with TODO: implement formatCode()
│       └── tests/
│           ├── run_suite.sh    # Agent-facing test runner
│           └── cases/          # ~200 test cases (generated by generate_cases.py)
├── solution/
│   └── solve.sh                # Oracle solution (uses reference formatter as backend)
└── tests/
    ├── test.sh                 # Verifier (runs original + hidden cases)
    ├── case_mapping.json       # Maps case numbers to source files
    └── hidden_cases/           # Variant cases (not visible to agent)
```

## Why These Are Hard

1. **No parser shortcuts** - Agents must build a real tokenizer + AST, not regex hacks
2. **Byte-exact matching** - Every space, newline, and trailing comma must be identical
3. **Ambiguous syntax** - Each language has cases where formatting depends on semantics
4. **Line-wrapping heuristics** - Must solve an optimization problem to match reference
5. **Idempotency** - Output must be stable (formatting twice produces same result)
6. **Comment/string preservation** - Content must survive while surrounding whitespace changes

## Language-Specific Challenges

### Go (`gofmt`)
- Tab indentation (not spaces)
- Automatic semicolon insertion after final token on line
- Operator spacing disambiguation (`a < -b` vs `a<-b`)
- Alignment of consecutive declarations
- Composite literal trailing commas

### Rust (`rustfmt`)
- Lifetime syntax (`&'a mut T` vs `& 'a mut T`)
- Macro formatting (`macro_rules!`, derive macros)
- Trait bounds and where clauses
- 4-space indent, 100-char line limit
- Method chain breaking at `.`

### Python (`black`)
- Indentation is syntactically significant
- Magic trailing comma controls line wrapping
- f-string internal preservation
- String quote normalization (single -> double)
- 88-char line limit (non-standard)
- Implicit line continuation rules

### C (`clang-format`)
- Pointer alignment (`int *p` not `int* p`)
- Preprocessor directive indentation
- K&R brace style
- 2-space indent
- Function pointer typedefs
- Macro expansion formatting

### Java (`google-java-format`)
- Lambda formatting and line breaking
- Generic type wildcards (`? extends T`)
- Annotation placement (one per line vs inline)
- Method chain breaking
- Import ordering (Android-style)
- 2-space indent, 100-char line limit

## How to Generate Test Cases

Each task has a `generate_cases.py` that:
1. Collects source files from the language's standard library / installed packages
2. Scores files by formatting interest (diverse syntax, moderate length)
3. Runs the reference formatter to produce expected outputs
4. Creates hidden variants by modifying strings/numbers/comments, then re-formats
5. Writes ~200 original cases + ~200 hidden variant cases

```bash
# Generate cases for Go formatter
cd code-formatter-go-go
python3 generate_cases.py

# Generate cases for all tasks
for task in code-formatter-go-go code-formatter-rust-rs code-formatter-py-py code-formatter-c-c code-formatter-java-java; do
    cd $task && python3 generate_cases.py && cd ..
done
```

## How to Run

```bash
# Install Harbor
uv tool install harbor

# Set API key
export ANTHROPIC_API_KEY=sk-...

# Run a single task
harbor run -p code-formatter-go-go -a claude-code -m anthropic/claude-opus-4-1

# Run all 5 tasks
for task in code-formatter-go-go code-formatter-rust-rs code-formatter-py-py code-formatter-c-c code-formatter-java-java; do
    harbor run -p $task -a claude-code -m anthropic/claude-opus-4-1
done

# Quick sanity check with oracle solution
harbor run -p code-formatter-go-go -k 1
```

## Expected Results

Based on the existing `code-formatter-js-js` task (Claude Opus 4.6: 38% pass rate), we expect:

| Model | Expected Pass Rate | Notes |
|-------|-------------------|-------|
| Claude Opus 4.6 | 30-45% | Best current model, still struggles with exact formatting |
| Claude Sonnet 4 | 20-35% | Less capable at complex parsing |
| GPT-4o | 15-30% | Weaker at deterministic formatting tasks |
| Codex CLI | 25-40% | Good at code, but byte-exact matching is hard |

## Reward Function

```
reward = passed_cases / total_cases
```

- Byte-for-byte comparison (diff -q)
- 30-second timeout per case
- Crashes and timeouts count as failures
- Both original + hidden cases are evaluated
